Onboard a New Restaurant

Follow this step-by-step guide to deploy a fully functional restaurant ordering system. The entire process takes about 30–60 minutes.

Firebase Firestore Vanilla JS AES Encrypted Mobile First ~45 min setup

How this worksThe system is based on The Cafe Darbar (/TCD/). You will copy that directory, rename it, and swap out config, branding, and menu data. All app logic remains unchanged.

Firebase Project Setup
Create the backend — Firestore database for orders, config, and admin history

Do this firstYou need the Firebase credentials before you can build credentials.json. Complete this step entirely before moving on.

Firebase Setup Checklist
Recommended Firestore Rules
Firestore Rules 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Config: public read, no public write
    match /config/{document} {
      allow read: if true;
      allow write: if false;
    }
    // Orders: anyone can write (place order), no public read
    match /orders/{document} {
      allow read:  if false;
      allow write: if true;
    }
    // Admin collections: locked down
    match /admin/{document} {
      allow read, write: if false;
    }
  }
}

4
Create restaurant.js
The single source of truth for all restaurant-specific config

Why this file mattersEvery script in the app reads from RESTAURANT object. Change this one file and the entire app adapts. Never hardcode restaurant values anywhere else.

MFC/restaurant.js 
const RESTAURANT = {
    name:           "My Food Corner",         // Full name shown in UI & WhatsApp messages
    prefix:         "mfc",                   // Short prefix — ALL localStorage keys use this
    encKey:         ['MFC', 'FOOD', 'CORNER'].join('-'),  // AES decrypt key for credentials.json
    logo:           "mfc-logo.png",           // Logo filename in this directory
    mapsUrl:        "https://maps.app.goo.gl/YOURLINK",  // Google Maps share link
    wpFallback:     "+919XXXXXXXXX",          // Fallback WhatsApp if Firestore unreachable
    minOrder:       200,                      // Minimum order for free delivery (₹)
    deliveryCharge: 50                       // Delivery charge below minimum (₹)
};

function lsKey(key) { return RESTAURANT.prefix + '_' + key; }

 Data Rules

                
    
                    
  • 🔑
    IDs must be unique integers. Duplicate IDs break the disable/enable items feature in admin.
    • 
                    
  • 🏷️
    Dish names must match image filenames. Veg Burger → image must be /food_src/Veg_Burger.webp (spaces → underscores).
    • 
                    
  • 🥗
    type must be exactly "Veg" or "NonVeg" — case-sensitive. This drives the veg/non-veg toggle filter.
    • 
                    
  • 💰
    price is a number (no ₹ symbol, no quotes). Decimals are allowed: 49.50.
    • 
                    
  • 📋
    Multiple subcategories can share the same name (e.g. "Starter" Veg and "Starter" NonVeg) — they display as separate sections.
    •

Without this, the app will fail silently. On first load, app.js reads shop status, opening/closing times, WhatsApp number, and disabled items from Firestore. If the document doesn't exist, the shop appears closed and WhatsApp falls back to wpFallback.

  • 4️⃣
    Add these fields to the document (see table below)

 10B · Patch cart.js to mirror orders into BankiBites

                
For BankiBites admin to see this restaurant's orders, every successful order must also be written into the central bankibites_orders collection. Three small edits to /MFC/cart.js:


                

                    

                        /MFC/cart.js — add to imports at the very top
                        
                    

                    
import { mirrorToBankiBites } from '../banki-bites/order-mirror.js';

                


                

                    

                        /MFC/cart.js — add this helper above collect_data()
                        
                    

                    
function flattenCartForMirror(cartItems) {
    const out = [];
    (cartItems || []).forEach(cat => {
        (cat.category?.dish_details || []).forEach(d => {
            out.push({ name: d.name, qty: d.quantity, price: d.price });
        });
    });
    return out;
}

                


                

                    

                        /MFC/cart.js — inside collect_data(), AFTER the addDoc resolves and localStorage push
                        
                    

                    
// Mirror to BankiBites central orders feed. Awaited so both tables are stable
// before the customer is sent to WhatsApp; failures are caught so the primary
// order is never lost even if the mirror is down.
let mirrored = false;
try {
    await mirrorToBankiBites({
        restaurant_id:   'MFC',                           // ← unique short code, UPPERCASE
        restaurant_name: RESTAURANT.name,
        source_doc_id:   docRef.id,                       // ← required; becomes mirror doc id mfc_<docRef.id>
        items:           flattenCartForMirror(data.order_details),
        subtotal:        cartTotalNumber - (data.delivery_charges || 0),
        total:           cartTotalNumber,
        delivery_charges: data.delivery_charges || 0,
        place:           data.mfc_place || '',            // ← match the <prefix>_place field you saved
        table_no:        data.table_no || '',
        source_doc_path: `${orderTable}/${docRef.id}`,
    });
    mirrored = true;
} catch (err) {
    console.warn('BankiBites mirror failed:', err);
}

                


                
    
                    
  • 🆔
    restaurant_id must be a unique, uppercase short code — used in the admin dashboard to identify the source. Pick something obvious (e.g. MFC for "My Food Corner").
    • 
                    
  • 📍
    place reads from data.<prefix>_place — the field name your data object uses (e.g. data.tcd_place, data.a1_place, data.mfc_place). Look for the matching line in collect_data() and use the same key.
    • 
                    
  • 🛡️
    The mirror is wrapped in .catch() so a BankiBites outage will NEVER block the restaurant's primary order flow — the customer still gets their WhatsApp order.
    • 
                

            


            
            

                
 10C · Verify Firestore rules

                
The order cascade (BankiBites admin marks delivered → restaurant order becomes Approved) requires the restaurant's order collection to allow status updates. The block you already pasted in step 1 covers this:

                

                    

                        firestore.rules — make sure this matches the new prefix
                        
                    

                    
match /mfc_orders/{docID} {
  allow create: if request.resource.data.keys().hasAll(['order_details', 'triggered_to', 'total_cart_value', 'table_no', 'status', 'created_at']);
  allow update: if request.resource.data.keys().hasAll(['status']);
}

                

                
No changes needed to bankibites_* rules — they already accept anonymous order creates with field-shape validation and only allow admin/delivery-staff to read/update.
